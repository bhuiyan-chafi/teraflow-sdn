# Execution and Processing of Optical Intents in ONOS

This report provides a comprehensive summary of how optical intents (such as those generated by the `jocn2023_intent_long.sh` script) are processed within the ONOS controller's architecture. It details the journey from the initial REST API request to the path computation, topology handling, and Routing and Spectrum Assignment (RSA), particularly emphasizing how parallel optical links are handled.

## 1. The HTTP REST Request and Intent Submission

The process begins when a script executes an HTTP POST request to the REST endpoint `/onos/optical/intents`.

* **REST Resource Handler**: The primary class listening to this URI is `OpticalIntentsWebResource.java` (located in `apps/optical-rest/...`).
* **Decoding**: Internal to this file, the `createIntent(InputStream stream)` method is triggered. It parses the incoming JSON, extracting crucial intent parameters such as the `ingressPoint` and `egressPoint`, the spectrum grid configuration (`signal`), and any constrained routes (`suggestedPath`).
* **Intent Instantiation**: The parsed components are handed off to `OpticalIntentUtility.java` via the `createExplicitOpticalIntent` method. This helper constructs an **`OpticalConnectivityIntent`** natively understood by the ONOS Intent Service and submits it to the controller.

## 2. Topology Generation and Context

The graph/topology representing the optical network is completely independent of the intent creation process.

* It is typically built ahead of time either dynamically (via southbound link-discovery using optical drivers) or injected manually using the ONOS `netcfg` (network configuration) subsystem.
* **Parallel Links Handling**: In the ONOS topology model, paths are rigorous sequences of `Link` objects. A topology edge (`Link`) is not just a node-to-node link; it's a strictly matched port-to-port connection (e.g., `Device A/Port 1` to `Device B/Port 1`). If there are multiple fiber pairs between the same two physical devices, they exist in the topology graph as entirely completely separate, distinct `Link` entities.

## 3. Path Computation

Once the `OpticalConnectivityIntent` is submitted, it lands in the intent compilation phase, specifically triggering the `OpticalConnectivityIntentCompiler.java`.

* **Suggested Path vs Algorithm Evaluation**: If a `suggestedPath` was supplied in the JSON (as seen in the `jocn2023_intent_long.sh` script), the compiler naturally bounds itself to use it.
* **K-Shortest Paths Routine**: If a path needs to be computed dynamically, the compiler references the `topologyService` and invokes `getOpticalPaths(...)`. This triggers an algorithm calculating K-Shortest Paths using custom link weight profiles (`LinkWeigher`) that exclude inactive links and adhere to matching port annotations.
* **Parallel Links in Pathing**: Because parallel links occupy uniquely distinct ports, the K-Shortest Path topological algorithm outputs each parallel link permutation as an entirely independent, distinct `Path` sequence object rather than treating them as a single pipe of aggregated bandwidth.

## 4. Routing and Spectrum Assignment (RSA)

The RSA stage maps to the compiler loop ensuring that spectrum continuity is satisfied end-to-end. This is executed by evaluating the `Paths` delivered by the preceding step.

* **Sequential Evaluation**: RSA applies strict isolation between independent paths. The compiler executes a loop (`for (Path path : paths)`) iterating sequentially over the available route permutations.
* **Spatial and Spectral Binding**: For a specific path iteration, `convertToResources` restricts the route into an explicit `List<Resource>` mapping exactly the source ports and requested `OchSignal` granularities (channel spacing, DWDM or FLEX grid properties).
* **Parallel Check behavior**: ONOS **does not** combine parallel links to evaluate gross bandwidth. Instead, it tests the spectrum continuous footprint along one specific `Path` utilizing one specific parallel link sequence via `allocateResourcesAttempt()`.
  * If that link undergoes wavelength contention, the `ResourceService` rejects the allocation.
  * The loop smoothly ignores it and evaluates the next topological `Path` which traverses the alternate parallel link, re-running the allocation check natively from scratch.
  * Once an allocation naturally succeeds, it breaks the loop and secures the sequence via `findFirstAvailableLambda()`.

## 5. Execution and Driver Push

Once RSA correctly assigns uninterrupted wavelength slots, the compiler converts the logic into network-installable `FlowRule` components. These optical rules are finally transmitted directly to the switch hardware via southbound OpenConfig mappings or NETCONF profiles within Device Drivers (e.g., `TerminalDeviceFlowRule.java` for ODTN models or `LumentumFlowRule.java` for direct ROADMs).
